Yes, you can preserve the aspect ratio when resizing images in Go by using the technique you described—detecting the original dimensions and ensuring the output maintains the same proportions. The standard approach is to set one dimension (width or height) to 0, which tells most image processing libraries to calculate the missing dimension automatically.

Here are two practical ways to implement this:

🖼️ Method 1: Preserve Aspect Ratio with imaging Library
The imaging package is the most popular choice for this purpose. When you set either width or height to 0, it automatically preserves the aspect ratio.

go
package main

import (
    "github.com/disintegration/imaging"
    "log"
    "os"
)

func main() {
    // Open the source image
    src, err := imaging.Open("input.jpg")
    if err != nil {
        log.Fatalf("failed to open image: %v", err)
    }

    // Get original dimensions
    srcBounds := src.Bounds()
    originalWidth := srcBounds.Dx()
    originalHeight := srcBounds.Dy()
    log.Printf("Original size: %dx%d", originalWidth, originalHeight)

    // Resize to width 800px while preserving aspect ratio (height = 0)
    dst := imaging.Resize(src, 800, 0, imaging.Lanczos)

    // Get new dimensions
    newBounds := dst.Bounds()
    log.Printf("New size: %dx%d", newBounds.Dx(), newBounds.Dy())

    // Save the result
    err = imaging.Save(dst, "output.jpg")
    if err != nil {
        log.Fatalf("failed to save image: %v", err)
    }
}

🔧 Method 2: Manual Aspect Ratio Calculation
If you need more control or your library doesn't support automatic aspect ratio preservation, you can calculate the dimensions manually:

go
package main

import (
    "image"
    "log"
    "os"

    _ "image/jpeg"
    _ "image/png"
)

func calculateAspectPreservingDimensions(originalWidth, originalHeight, targetWidth, targetHeight int) (int, int) {
    if targetWidth == 0 && targetHeight == 0 {
        return originalWidth, originalHeight
    }

    if targetWidth == 0 {
        // Height is fixed, calculate width
        ratio := float64(targetHeight) / float64(originalHeight)
        return int(float64(originalWidth) * ratio), targetHeight
    }

    if targetHeight == 0 {
        // Width is fixed, calculate height
        ratio := float64(targetWidth) / float64(originalWidth)
        return targetWidth, int(float64(originalHeight) * ratio)
    }

    // Both dimensions specified — use the smaller scale to fit within box
    widthRatio := float64(targetWidth) / float64(originalWidth)
    heightRatio := float64(targetHeight) / float64(originalHeight)
    
    if widthRatio < heightRatio {
        return targetWidth, int(float64(originalHeight) * widthRatio)
    }
    return int(float64(originalWidth) * heightRatio), targetHeight
}

func main() {
    // Open and decode image
    file, err := os.Open("input.jpg")
    if err != nil {
        log.Fatal(err)
    }
    defer file.Close()

    img, _, err := image.Decode(file)
    if err != nil {
        log.Fatal(err)
    }

    bounds := img.Bounds()
    originalWidth, originalHeight := bounds.Dx(), bounds.Dy()
    log.Printf("Original: %dx%d", originalWidth, originalHeight)

    // Calculate new dimensions — width fixed to 800, height auto-calculated
    newWidth, newHeight := calculateAspectPreservingDimensions(originalWidth, originalHeight, 800, 0)
    log.Printf("New dimensions: %dx%d", newWidth, newHeight)

    // You would then use these dimensions with your chosen resizing function
}

I want to run concurrent (gorountinue and channel) and sequential with go - I want the program to be able to run with or without. How do I do that?
Great question! You can design your program to support both concurrent (goroutines + channels) and sequential execution using a strategy pattern or a simple flag/configuration. Here are several approaches:

🎯 Method 1: Simple Flag-Based Approach (Most Common)
go
package main

import (
    "flag"
    "fmt"
    "log"
    "sync"
    "time"
)

// Image processing task
type ImageTask struct {
    ID       int
    FilePath string
    Result   string
    Error    error
}

func processImage(task ImageTask) ImageTask {
    // Simulate image processing work
    time.Sleep(100 * time.Millisecond)
    task.Result = fmt.Sprintf("Processed %s", task.FilePath)
    return task
}

// Sequential processing
func processSequential(tasks []ImageTask) []ImageTask {
    results := make([]ImageTask, len(tasks))
    for i, task := range tasks {
        results[i] = processImage(task)
        fmt.Printf("Sequential: Completed task %d\n", task.ID)
    }
    return results
}

// Concurrent processing using goroutines and channels
func processConcurrent(tasks []ImageTask) []ImageTask {
    results := make([]ImageTask, len(tasks))
    resultChan := make(chan ImageTask, len(tasks))
    var wg sync.WaitGroup

    // Launch goroutines
    for _, task := range tasks {
        wg.Add(1)
        go func(t ImageTask) {
            defer wg.Done()
            processed := processImage(t)
            resultChan <- processed
            fmt.Printf("Concurrent: Completed task %d\n", t.ID)
        }(task)
    }

    // Close channel when all goroutines complete
    go func() {
        wg.Wait()
        close(resultChan)
    }()

    // Collect results
    i := 0
    for result := range resultChan {
        results[i] = result
        i++
    }

    return results
}

func main() {
    // Command line flag to switch between modes
    concurrent := flag.Bool("concurrent", false, "Run with goroutines and channels")
    numTasks := flag.Int("tasks", 10, "Number of tasks to process")
    flag.Parse()

    // Create tasks
    tasks := make([]ImageTask, *numTasks)
    for i := 0; i < *numTasks; i++ {
        tasks[i] = ImageTask{
            ID:       i,
            FilePath: fmt.Sprintf("image_%d.jpg", i),
        }
    }

    // Choose processing mode
    var results []ImageTask
    start := time.Now()

    if *concurrent {
        fmt.Println("Running in CONCURRENT mode with goroutines and channels")
        results = processConcurrent(tasks)
    } else {
        fmt.Println("Running in SEQUENTIAL mode")
        results = processSequential(tasks)
    }

    elapsed := time.Since(start)
    fmt.Printf("Completed %d tasks in %v\n", len(results), elapsed)
}

go
package main

import (
    "fmt"
    "runtime"
    "sync"
    "time"
)

// Processor interface defines the strategy
type ImageProcessor interface {
    Process(images []Image) []ProcessedImage
    GetName() string
}

// Data structures
type Image struct {
    ID   int
    Data string
}

type ProcessedImage struct {
    ImageID int
    Result  string
}

// Sequential processor
type SequentialProcessor struct{}

func (s SequentialProcessor) Process(images []Image) []ProcessedImage {
    results := make([]ProcessedImage, len(images))
    
    for i, img := range images {
        results[i] = s.processImage(img)
        fmt.Printf("[Sequential] Processed image %d\n", img.ID)
    }
    
    return results
}

func (s SequentialProcessor) processImage(img Image) ProcessedImage {
    // Simulate work
    time.Sleep(50 * time.Millisecond)
    return ProcessedImage{
        ImageID: img.ID,
        Result:  fmt.Sprintf("Processed: %s", img.Data),
    }
}

func (s SequentialProcessor) GetName() string {
    return "Sequential Processor"
}

// Concurrent processor with goroutines and channels
type ConcurrentProcessor struct {
    workerCount int
}

func NewConcurrentProcessor(workers int) *ConcurrentProcessor {
    if workers == 0 {
        workers = runtime.NumCPU()
    }
    return &ConcurrentProcessor{workerCount: workers}
}

func (c ConcurrentProcessor) Process(images []Image) []ProcessedImage {
    type workItem struct {
        index int
        image Image
    }
    
    type resultItem struct {
        index  int
        result ProcessedImage
    }
    
    // Create channels
    workChan := make(chan workItem, len(images))
    resultChan := make(chan resultItem, len(images))
    
    // Start worker goroutines
    var wg sync.WaitGroup
    for w := 0; w < c.workerCount; w++ {
        wg.Add(1)
        go c.worker(workChan, resultChan, &wg, w)
    }
    
    // Send work to channel
    for i, img := range images {
        workChan <- workItem{index: i, image: img}
    }
    close(workChan)
    
    // Wait for workers and close result channel
    go func() {
        wg.Wait()
        close(resultChan)
    }()
    
    // Collect results
    results := make([]ProcessedImage, len(images))
    for res := range resultChan {
        results[res.index] = res.result
        fmt.Printf("[Concurrent] Collected result for image %d\n", res.result.ImageID)
    }
    
    return results
}

func (c ConcurrentProcessor) worker(workChan <-chan workItem, resultChan chan<- resultItem, wg *sync.WaitGroup, workerID int) {
    defer wg.Done()
    
    for work := range workChan {
        // Process the image
        processed := ProcessedImage{
            ImageID: work.image.ID,
            Result:  fmt.Sprintf("Processed by worker %d: %s", workerID, work.image.Data),
        }
        time.Sleep(50 * time.Millisecond) // Simulate work
        
        resultChan <- resultItem{
            index:  work.index,
            result: processed,
        }
        fmt.Printf("[Concurrent] Worker %d processed image %d\n", workerID, work.image.ID)
    }
}

func (c ConcurrentProcessor) GetName() string {
    return fmt.Sprintf("Concurrent Processor (%d workers)", c.workerCount)
}

// Context that uses the processor strategy
type ProcessingContext struct {
    processor ImageProcessor
}

func (pc *ProcessingContext) SetProcessor(p ImageProcessor) {
    pc.processor = p
}

func (pc *ProcessingContext) Execute(images []Image) []ProcessedImage {
    fmt.Printf("\nUsing: %s\n", pc.processor.GetName())
    return pc.processor.Process(images)
}

func main() {
    // Create test data
    images := make([]Image, 20)
    for i := range images {
        images[i] = Image{ID: i, Data: fmt.Sprintf("Image data %d", i)}
    }
    
    // Sequential processing
    context := &ProcessingContext{}
    context.SetProcessor(SequentialProcessor{})
    
    start := time.Now()
    results1 := context.Execute(images)
    fmt.Printf("Sequential completed in %v\n\n", time.Since(start))
    
    // Concurrent processing with goroutines and channels
    context.SetProcessor(NewConcurrentProcessor(5))
    
    start = time.Now()
    results2 := context.Execute(images)
    fmt.Printf("Concurrent completed in %v\n", time.Since(start))
    
    _ = results1
    _ = results2
}
🔄 Method 3: Configurable Pipeline with Worker Pool
This approach is great for production systems where you need fine-grained control:

go
package main

import (
    "context"
    "flag"
    "fmt"
    "log"
    "sync"
    "time"
)

type ImageProcessor struct {
    // Configuration
    concurrent bool
    workers    int
    batchSize  int
    
    // Internal
    mu sync.Mutex
}

type ProcessConfig struct {
    Concurrent bool
    Workers    int
    BatchSize  int
}

func NewImageProcessor(config ProcessConfig) *ImageProcessor {
    if config.Workers == 0 {
        config.Workers = 4
    }
    if config.BatchSize == 0 {
        config.BatchSize = 10
    }
    
    return &ImageProcessor{
        concurrent: config.Concurrent,
        workers:    config.Workers,
        batchSize:  config.BatchSize,
    }
}

// ProcessImages is the main entry point that decides which mode to use
func (ip *ImageProcessor) ProcessImages(images []string) error {
    if ip.concurrent {
        return ip.processConcurrent(images)
    }
    return ip.processSequential(images)
}

// Sequential processing
func (ip *ImageProcessor) processSequential(images []string) error {
    log.Printf("Processing %d images sequentially", len(images))
    
    for i, img := range images {
        if err := ip.processImage(img, i); err != nil {
            return fmt.Errorf("failed to process image %s: %w", img, err)
        }
    }
    
    return nil
}

// Concurrent processing with worker pool
func (ip *ImageProcessor) processConcurrent(images []string) error {
    log.Printf("Processing %d images concurrently with %d workers", len(images), ip.workers)
    
    // Create channels
    jobs := make(chan job, len(images))
    results := make(chan error, len(images))
    
    // Create worker pool
    var wg sync.WaitGroup
    for w := 0; w < ip.workers; w++ {
        wg.Add(1)
        go ip.worker(w, jobs, results, &wg)
    }
    
    // Send jobs
    for i, img := range images {
        jobs <- job{index: i, imagePath: img}
    }
    close(jobs)
    
    // Wait for workers and close results
    go func() {
        wg.Wait()
        close(results)
    }()
    
    // Collect results
    for err := range results {
        if err != nil {
            return err
        }
    }
    
    return nil
}

type job struct {
    index     int
    imagePath string
}

func (ip *ImageProcessor) worker(id int, jobs <-chan job, results chan<- error, wg *sync.WaitGroup) {
    defer wg.Done()
    
    for job := range jobs {
        log.Printf("Worker %d processing image %d: %s", id, job.index, job.imagePath)
        
        err := ip.processImage(job.imagePath, job.index)
        results <- err
    }
}

// Actual image processing logic (shared between both modes)
func (ip *ImageProcessor) processImage(imagePath string, index int) error {
    // Simulate work with lock for thread-safe logging
    ip.mu.Lock()
    log.Printf("Processing: %s (index: %d)", imagePath, index)
    ip.mu.Unlock()
    
    // Simulate actual image processing time
    time.Sleep(100 * time.Millisecond)
    
    // Simulate occasional error
    if index == 42 {
        return fmt.Errorf("simulated error on image %d", index)
    }
    
    return nil
}

// With context cancellation support
func (ip *ImageProcessor) ProcessWithContext(ctx context.Context, images []string) error {
    if ip.concurrent {
        return ip.processConcurrentWithContext(ctx, images)
    }
    return ip.processSequentialWithContext(ctx, images)
}

func (ip *ImageProcessor) processConcurrentWithContext(ctx context.Context, images []string) error {
    jobs := make(chan job, len(images))
    results := make(chan error, len(images))
    
    var wg sync.WaitGroup
    for w := 0; w < ip.workers; w++ {
        wg.Add(1)
        go ip.workerWithContext(ctx, w, jobs, results, &wg)
    }
    
    // Send jobs with context check
    for i, img := range images {
        select {
        case <-ctx.Done():
            return ctx.Err()
        case jobs <- job{index: i, imagePath: img}:
        }
    }
    close(jobs)
    
    go func() {
        wg.Wait()
        close(results)
    }()
    
    // Collect results with context
    for err := range results {
        select {
        case <-ctx.Done():
            return ctx.Err()
        default:
            if err != nil {
                return err
            }
        }
    }
    
    return nil
}

func (ip *ImageProcessor) workerWithContext(ctx context.Context, id int, jobs <-chan job, results chan<- error, wg *sync.WaitGroup) {
    defer wg.Done()
    
    for {
        select {
        case <-ctx.Done():
            results <- ctx.Err()
            return
        case job, ok := <-jobs:
            if !ok {
                return
            }
            err := ip.processImage(job.imagePath, job.index)
            results <- err
        }
    }
}

func (ip *ImageProcessor) processSequentialWithContext(ctx context.Context, images []string) error {
    for i, img := range images {
        select {
        case <-ctx.Done():
            return ctx.Err()
        default:
            if err := ip.processImage(img, i); err != nil {
                return err
            }
        }
    }
    return nil
}

func main() {
    // Command line flags
    concurrent := flag.Bool("concurrent", false, "Enable concurrent processing")
    workers := flag.Int("workers", 4, "Number of workers for concurrent mode")
    timeout := flag.Duration("timeout", 30*time.Second, "Timeout for processing")
    
    flag.Parse()
    
    // Create test images
    images := make([]string, 20)
    for i := range images {
        images[i] = fmt.Sprintf("image_%d.jpg", i)
    }
    
    // Create processor
    processor := NewImageProcessor(ProcessConfig{
        Concurrent: *concurrent,
        Workers:    *workers,
    })
    
    // Create context with timeout
    ctx, cancel := context.WithTimeout(context.Background(), *timeout)
    defer cancel()
    
    // Process images
    start := time.Now()
    err := processor.ProcessWithContext(ctx, images)
    elapsed := time.Since(start)
    
    if err != nil {
        log.Printf("Error: %v", err)
    }
    
    log.Printf("Processing completed in %v", elapsed)
}

After reviewing industry practices and community standards, the Worker Pool pattern using goroutines and channels is the industry standard approach for running configurable concurrent and sequential operations in Go .

This pattern provides the right balance of performance, resource control, and code clarity that production systems demand.

🏭 Why the Worker Pool is Industry Standard
Here's what makes the Worker Pool pattern the go-to solution:

Requirement	How Worker Pool Solves It
Bounded concurrency	Limits goroutines to a fixed number (e.g., runtime.NumCPU()), preventing resource exhaustion 
Sequential mode	Simply set worker count to 1
Concurrent mode	Set worker count > 1
Backpressure handling	Buffered task channels block producers when full, preventing unbounded memory growth 
Clean shutdown	sync.WaitGroup ensures all workers complete before exiting 
Error propagation	Results channel collects errors without losing them
Cancellation support	context.Context enables timeout and graceful shutdown

Here's the industry-standard Worker Pool implementation used in production Go systems :

go
package main

import (
    "context"
    "flag"
    "fmt"
    "log"
    "runtime"
    "sync"
    "time"
)

// Config holds the processing configuration
type Config struct {
    Concurrent bool
    Workers    int
    Timeout    time.Duration
}

// Task represents a unit of work
type Task struct {
    ID   int
    Data string
}

// Result represents the outcome of a task
type Result struct {
    TaskID int
    Output string
    Err    error
}

// WorkerPool is the industry-standard pattern for concurrent processing
type WorkerPool struct {
    config  Config
    workers int
}

func NewWorkerPool(config Config) *WorkerPool {
    workers := config.Workers
    if workers == 0 {
        workers = runtime.NumCPU()
    }
    return &WorkerPool{
        config:  config,
        workers: workers,
    }
}

// Process executes tasks either sequentially or concurrently based on config
func (wp *WorkerPool) Process(ctx context.Context, tasks []Task) ([]Result, error) {
    if wp.config.Concurrent {
        return wp.processConcurrent(ctx, tasks)
    }
    return wp.processSequential(ctx, tasks)
}

// processSequential runs tasks one after another
func (wp *WorkerPool) processSequential(ctx context.Context, tasks []Task) ([]Result, error) {
    results := make([]Result, 0, len(tasks))
    
    for i, task := range tasks {
        select {
        case <-ctx.Done():
            return nil, ctx.Err()
        default:
            result := wp.executeTask(task)
            results = append(results, result)
            if result.Err != nil {
                log.Printf("Task %d failed in sequential mode: %v", task.ID, result.Err)
            }
        }
        _ = i
    }
    
    return results, nil
}

// processConcurrent runs tasks using the worker pool pattern
// This is the industry-standard implementation [citation:2][citation:7]
func (wp *WorkerPool) processConcurrent(ctx context.Context, tasks []Task) ([]Result, error) {
    if len(tasks) == 0 {
        return []Result{}, nil
    }
    
    // Create channels for task distribution and result collection
    taskChan := make(chan Task, len(tasks))
    resultChan := make(chan Result, len(tasks))
    
    // Worker pool with WaitGroup for clean shutdown [citation:2][citation:8]
    var wg sync.WaitGroup
    
    // Start workers
    for w := 0; w < wp.workers; w++ {
        wg.Add(1)
        go wp.worker(ctx, w, taskChan, resultChan, &wg)
    }
    
    // Send all tasks to the channel
    for _, task := range tasks {
        select {
        case <-ctx.Done():
            return nil, ctx.Err()
        case taskChan <- task:
        }
    }
    close(taskChan)
    
    // Close result channel when all workers complete
    go func() {
        wg.Wait()
        close(resultChan)
    }()
    
    // Collect results
    results := make([]Result, 0, len(tasks))
    for result := range resultChan {
        results = append(results, result)
    }
    
    return results, nil
}

// worker processes tasks from the task channel [citation:2]
func (wp *WorkerPool) worker(ctx context.Context, id int, tasks <-chan Task, results chan<- Result, wg *sync.WaitGroup) {
    defer wg.Done()
    
    for task := range tasks {
        select {
        case <-ctx.Done():
            results <- Result{TaskID: task.ID, Err: ctx.Err()}
            return
        default:
            result := wp.executeTask(task)
            results <- result
        }
    }
}

}

    defer cancel()
    
    // Process with automatic mode selection
    log.Printf("Starting processing with concurrent=%v", *concurrent)
    start := time.Now()
    
    results, err := ProcessImages(ctx, images, *concurrent)
    
    elapsed := time.Since(start)
    
    if err != nil {
        log.Fatalf("Processing failed: %v", err)
    }
    
    // Count successes and failures
    successCount := 0
    for _, r := range results {
        if r.Err == nil {
            successCount++
        }
    }
    
    log.Printf("Completed %d/%d tasks in %v", successCount, len(results), elapsed)
}
🎯 How to Switch Between Modes
The beauty of this industry-standard approach is how simple the switching becomes:

bash
# Sequential mode (single worker, deterministic execution)
go run main.go

# Concurrent mode (N workers running in parallel)
go run main.go -concurrent

# With custom task count
go run main.go -concurrent -tasks=100
📚 Why This Pattern Dominates Go Production Code
After analyzing the industry landscape, here's why the Worker Pool pattern is standard:

1. Battle-Tested in Production
Major Go projects (Docker, Kubernetes, Terraform) use variations of this pattern for task processing . Google's internal benchmarks use a similar worker pool implementation .

2. Natural Fit with Go's Philosophy
"Don't communicate by sharing memory; share memory by communicating" - Channels embody this principle 

The pattern uses goroutines (lightweight threads) efficiently with runtime.NumCPU() 

select statements provide elegant cancellation and timeout handling 

3. Built on Official Primitives
The pattern uses:

context.Context - Official cancellation API 

sync.WaitGroup - Standard goroutine coordination 

Channels - First-class concurrency primitive 

4. Prevents Critical Issues
Issue	How Worker Pool Prevents It
Goroutine leaks	WaitGroup ensures all workers terminate 
Resource exhaustion	Fixed worker count limits concurrency 
Race conditions	Channel-based communication eliminates shared state 
Unbounded memory	Buffered channels provide backpressure
